# Ejercicio 6.2 — Forzado y bloqueo de herencia (RA7.b)

Basado en la teoría de clases abstractas, interfaces y control de herencia (Unidad 6.1).

## Objetivo

Partiendo de una versión base **sin** clase abstracta (v0), implementar una versión final que:

- use **una clase abstracta con sentido**,
- aplique mecanismos para **forzar** y **bloquear** la herencia/métodos,
- y mantenga el programa funcionando (con “logs” por consola).

## Entregables en este repositorio

- **Código Kotlin**:
  - Versión base (v0): `es.ies.ejercicios.u6.ej62.v0`
  - Versión final (con clase abstracta): `es.ies.ejercicios.u6.ej62`
- **Documentación**: este fichero `docs/ejercicios/6.2.md`

## I. Práctica

### I.a) Punto de partida (v0) y problema detectado

**RESPUESTA:**  
En la versión v0 existe un `ReportGeneratorV0` que decide el formato con un `when` según un `enum` (`ReportFormatV0`).  
El problema típico es que, si añadimos más formatos (HTML, JSON, etc.), el `when` crecería y la clase central acumularía responsabilidades (cabecera, formato de línea, etc.), dificultando la extensibilidad.


---

### I.b) Versión final (con clase abstracta): diseño aplicado

#### Clase abstracta (“Template Method”)

**RESPUESTA:**  
Se crea una clase abstracta `ReportTemplate` que define el **algoritmo común** para generar un informe:

1) Construir cabecera (`header(title)`)  
2) Formatear cada línea (`formatLine(line)`)  
3) Añadir pie (`footer()`)

La idea es separar **lo común** (el orden y estructura del algoritmo) de **lo variable** (cómo se representa cada línea/cabecera en cada formato).


#### Subclases concretas

**RESPUESTA:**  
Se implementan al menos dos subclases:

- `CsvReport`: implementa la cabecera y el formato de cada línea en CSV.
- `MarkdownReport`: implementa la cabecera y el formato de cada línea en Markdown.

Así, al añadir un formato nuevo, se añade una nueva subclase sin modificar el algoritmo común (evitamos el `when` creciente de v0).

**Enlaces permanentes (subclases):**
- `CsvReport`: PON AQUÍ TU PERMALINK  
- `MarkdownReport`: PON AQUÍ TU PERMALINK  

---

### I.c) Sustituir selección por polimorfismo

**RESPUESTA:**  
En el `main` de la versión final, en lugar de usar un `when`/`enum` para seleccionar el formato, se trabaja con una lista de `ReportTemplate`:

- Tipo estático (padre): `ReportTemplate`
- Tipos reales (hijos): `CsvReport`, `MarkdownReport`

Al llamar a `generate(...)` se ejecuta el algoritmo común (bloqueado) y cada subclase aporta su implementación de `formatLine(...)` (forzada) y, si procede, `header(...)`/`footer()`.

---

### I.d) Interfaz añadida (rol transversal)

**RESPUESTA:**  
Se usa una `fun interface Logger` para representar un rol transversal de “registro de mensajes”.  
Tiene sentido usar interfaz aquí porque:

- No necesitamos una jerarquía de “tipos de logger” con estado compartido obligatorio.
- Queremos permitir múltiples implementaciones sin forzar herencia de clases (ej. stdout, fichero, nulo, etc.).
- Facilita inyección por composición: el `main` puede depender de `Logger` y no de una clase concreta.

## II. Teoría

### II.a) ¿Qué beneficios obtienes al usar una clase abstracta? ¿Y de una interface?

**RESPUESTA:**
- **Clase abstracta (`ReportTemplate`)**:
  - Permite **reutilizar** el algoritmo común (template method) sin duplicarlo en cada formato.
  - Permite **forzar** pasos obligatorios (`abstract fun formatLine(...)`).
  - Facilita el polimorfismo: el código cliente trabaja con el tipo padre (`ReportTemplate`) y cambia el comportamiento por subclase.
- **Interfaz (`Logger`)**:
  - Define un “contrato” independiente de una jerarquía de clases.
  - Permite cambiar implementaciones sin acoplar el código a una clase concreta.
  - Permite que una clase implemente múltiples interfaces (mejor que herencia múltiple, que Kotlin no permite en clases).

---

### II.b) ¿Qué modificadores y mecanismos has utilizado para bloquear y forzar la herencia de clases y métodos?

**RESPUESTA:**
- **Forzar (obligar a implementar):**
  - `abstract class ReportTemplate`
  - `protected abstract fun formatLine(line: String): String`  
  (cada subclase está obligada a implementar el formateo de línea)

- **Bloquear (impedir cambios no deseados):**
  - `final fun generate(...)` en `ReportTemplate` para impedir que una subclase altere el algoritmo común (orden de pasos).
  - `final override fun log(...)` en `StdoutLogger` para impedir que otra subclase cambie el comportamiento de logging ya “cerrado”.
